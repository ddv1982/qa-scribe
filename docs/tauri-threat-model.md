# Tauri Trust-Boundary Review

Date: 2026-07-12. Baseline: v0.7.10.

## Assets and trust boundaries

QA Scribe is a local-first, single-window desktop application. The WebView handles presentation and user interaction. Rust has unrestricted local-process, clipboard, app-data, and SQLite access, so every Tauri command is a privilege boundary even when its TypeScript caller is generated.

The application does not render remote application code, create remote windows, or intentionally embed third-party frames. Provider authentication stays in local provider CLIs. Session content, generated HTML, clipboard data, attachment paths, and provider output are untrusted inputs.

## Existing controls

- The main window receives one named capability. It grants the application `default` permission set plus only event listen/unlisten and window close.
- `build.rs` supplies an explicit `AppManifest::commands` allowlist.
- The Specta builder is the runtime invoke handler and the source of generated TypeScript bindings.
- Structured command errors cross the IPC boundary instead of backend strings being interpreted as error types.
- The CSP permits local assets, data/asset images needed by editor and clipboard flows, and inline styles; it does not permit remote scripts.
- Generated rich HTML is sanitized in the frontend and validated/sanitized again in the Rust generation path.
- Attachment reads are by managed attachment ID, and imported attachment content is validated before persistence.

`bun run tauri:commands:check` now proves that the build allowlist, Specta handler, generated bindings, default permission set, and autogenerated permission files contain exactly the same command names. It also proves the default capability targets the main window and includes the app permission set.

## Isolation Pattern decision

Do not enable Tauri's Isolation Pattern in the current single-window release.

The present application loads only bundled application code under a restrictive CSP and exposes a narrow, explicitly checked command surface. Isolation would add another message-processing context to editor, clipboard, and channel-based generation flows without eliminating the need for command-level authorization and input validation. The additional complexity is not justified by an identified remote-content or multi-principal use case today.

Revisit this decision before any of the following:

- loading remote application content or third-party frames;
- opening windows with different privilege levels;
- adding plugin permissions broader than the current main-window needs;
- accepting navigations away from the bundled frontend;
- exposing filesystem paths or generic process execution to the WebView.

## Required invariants

1. Every registered command appears in all five checked command-surface representations.
2. Capabilities remain scoped by window label, never title or wildcard without an ADR.
3. No command accepts an arbitrary filesystem path or executable from the WebView.
4. HTML/provider/clipboard inputs are treated as hostile at the Rust boundary even when the frontend sanitizes them.
5. New remote-content or multi-window behavior requires a renewed threat review and Isolation Pattern decision.
